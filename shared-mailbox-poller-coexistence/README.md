# Automated Email Poller Coexistence with Human Users in a Shared M365 Mailbox

## Executive Summary

An automated poller scanning a shared Exchange Online mailbox alongside human users faces several fundamental problems. The most critical: **the `\Seen` flag is globally shared**, so `IMAP SEARCH UNSEEN` will miss any email a staff member reads first. IMAP custom keyword flags are not supported by Exchange Online, and IMAP UIDs change when messages are moved between folders. The recommended architecture uses **Microsoft Graph API** rather than IMAP, with either a **transport rule copying mail to a dedicated processing mailbox** (strongest isolation) or **direct Graph API monitoring of the shared mailbox** using delta queries, immutable IDs, and categories as processed markers.

---

## 1. The UNSEEN/Read Flag Problem

### The `\Seen` Flag is Globally Shared

Exchange Online shared mailboxes maintain a **single, global read state**. The `\Seen` flag is shared across all IMAP sessions, Outlook desktop, Outlook Web Access (OWA), and Outlook mobile. There is **no per-user read state** on shared mailboxes.

**What this means in practice:**
- Staff member opens email in Outlook at 10:00:01 AM -> email is marked `\Seen`
- Poller runs `IMAP SEARCH UNSEEN` at 10:00:30 AM -> email is **not returned**
- The poller has permanently missed this email

This behavior is confirmed by Microsoft: the read flag is connected to the mailbox itself, not to individual users. If one user reads an email, all other users (and IMAP sessions) see it as read.

### Can You Have Per-User Read State?

**No**, not on a shared mailbox in Exchange Online. The alternatives that DO support per-user read state are:
- **Public Folders** (legacy, not recommended for new deployments)
- **Microsoft 365 Groups** (maintain per-user read counts)

Neither of these is a drop-in replacement for a shared mailbox in most organizational workflows.

### Verdict

**`IMAP SEARCH UNSEEN` is fundamentally broken for this use case.** Any architecture that relies on the `\Seen` flag for message discovery will have a race condition with human users. This is the single most common mistake in shared mailbox automation design.

## 2. Message Tracking Strategies That Don't Rely on Read Flags

### 2.1 IMAP UIDs

| Property | Behavior |
|----------|----------|
| Stability within a folder | Stable across sessions (should not change) |
| When message is moved | **UID changes** - new UID assigned in destination folder |
| UIDVALIDITY | Changes when UIDs have been reassigned; client must re-sync |
| Uniqueness | Unique within a folder + UIDVALIDITY combination only |

**Key limitation:** UIDs are folder-scoped. Moving a message from Inbox to a subfolder invalidates the old UID and assigns a new one. A poller tracking "highest UID seen" in the Inbox will correctly detect new arrivals but cannot follow messages that were moved before being scanned.

**Known Exchange issue:** There are reports of Exchange Online assigning different UIDs to the same email (same Message-ID, same UIDVALIDITY) in edge cases, particularly involving attachment size differences.

**Viable strategy with IMAP UIDs:**
- Track the highest UID seen in Inbox
- On each poll: `SEARCH UID {lastUID+1}:*`
- This catches new arrivals but **misses messages moved out of Inbox before scan**

### 2.2 IMAP UIDVALIDITY

UIDVALIDITY is a per-folder value that changes when the server has reassigned UIDs (e.g., after mailbox repair, migration, or store rebuild). When UIDVALIDITY changes:
- All cached UIDs for that folder are invalid
- Client must perform a full re-sync

In Exchange Online, UIDVALIDITY changes are rare but can happen during backend maintenance or mailbox moves between databases.

### 2.3 Message-ID Header (RFC 5322)

The `Message-ID` header is generated by the sending mail system and is **generally unique**, but with caveats:

- Some automated systems reuse the same Message-ID across messages
- Exchange's own duplicate detection uses Message-ID **combined with** client submit time
- Same Message-ID + same Date header will cause Exchange to treat messages as duplicates and potentially suppress delivery

**For this use case:** If the emails containing the identifier pattern are generated by a known system with proper Message-ID generation, the Message-ID is a reliable external tracking key. Store processed Message-IDs in an external database.

### 2.4 Microsoft Graph API: `id` vs `immutableId`

| Property | Behavior on folder move | Behavior on archive move | Shared mailbox support |
|----------|------------------------|-------------------------|----------------------|
| `id` (default) | **Changes** | Changes | Yes |
| `immutableId` | **Stable** | Changes | Yes (for messages) |

**How to use immutableId:**
- Include `Prefer: IdType="ImmutableId"` header in all Graph API requests
- Works with delta queries and change notifications
- Can translate between regular and immutable IDs via `translateExchangeIds`
- Supported resources: message, attachment, event, eventMessage, contact, outlookTask
- Container types (mailFolder) do NOT support immutableId, but their regular IDs are already constant

**Important:** immutableId changes if the item is moved to an **archive mailbox** or exported and re-imported. Within the same mailbox (including moves between folders), it remains stable.

### 2.5 Exchange Categories as "Processed" Markers

Categories are Exchange-native metadata visible in Outlook, OWA, and via Graph API.

**Via Graph API (recommended):**
```
PATCH /users/{sharedMailboxId}/messages/{messageId}
Content-Type: application/json

{
  "categories": ["Bot-Processed"]
}
```

**Via IMAP:** Not supported. Exchange Online's IMAP implementation does not expose categories, and the `PERMANENTFLAGS` response does not include `\*` (no custom keyword support).

**Pros:**
- Visible to staff in Outlook (transparent)
- Survives folder moves within the mailbox
- Can filter by category in Graph queries
- Simple to implement

**Cons:**
- Staff could accidentally remove the category
- Requires Graph API (not available via IMAP)
- Requires `MailboxSettings.ReadWrite` application permission for master category list management

### 2.6 Extended MAPI Properties / Open Extensions (Graph API)

**Open Extensions (recommended by Microsoft):**
- Flexible JSON-like custom data attached to a message
- Stored as MAPI named properties
- Invisible to Outlook users (not displayed in UI)
- Finite resource: MAPI named property quota per mailbox can be exhausted

**Extended Properties (legacy):**
- Direct MAPI property access
- Can filter messages: `$filter=singleValueExtendedProperties/Any(ep: ep/id eq '...' and ep/value eq '...')`
- Lower-level but more compatible with existing Exchange tooling

**Use case:** Stamp a custom property `ProcessedByBot=true` on each message. More robust than categories (staff cannot accidentally remove it) but invisible for troubleshooting.

## 3. The "Moved Before Scan" Problem

### Scenario: Staff Moves Email from Inbox Before Poller Sees It

This is the second most critical race condition after the read-flag problem.

**With IMAP polling Inbox only:**
- Email arrives in Inbox at UID 500
- Staff moves it to "Projects" folder at 10:00:05 AM
- Poller scans Inbox at 10:00:30 AM, searches UID 490:* -> UID 500 no longer exists in Inbox
- Email is **permanently missed**

**With IMAP polling Inbox + all subfolders:**
- Theoretically catches moved messages, but requires scanning every folder every cycle
- UID tracking becomes complex (different UID spaces per folder)
- Performance degrades with many folders
- Still misses deletions

### Scenario: Staff Deletes Email Before Poller Sees It

**With IMAP:** Deleted messages go to Deleted Items (or are purged). No mechanism to detect "message that was in Inbox but is now gone."

**With Litigation Hold:** Deleted items are preserved in Recoverable Items. This provides a safety net but requires Exchange Online Plan 2 and adds complexity to the poller.

### How Graph API Delta Queries Handle This

Delta queries are the correct solution for the "moved before scan" problem:

1. **Delta query is per-folder** - you must track each folder individually
2. **Moved/deleted items** return as `@removed` entries with `"reason": "deleted"`
3. **Using immutableId** - if you track by immutableId, you can detect that a message removed from the Inbox still exists in another folder
4. **Full mailbox sync** approach:
   - Use `mailFolder: delta` to track folder structure changes
   - Use `message: delta` on each folder to track message changes
   - Maintain a local index keyed by immutableId
   - When a message appears as `@removed` from Inbox, check if it appeared as `created` in another folder

**Practical delta query approach for a poller:**
```
GET /users/{sharedMailboxId}/mailFolders/Inbox/messages/delta
Prefer: IdType="ImmutableId"
```
- First call returns all current messages + deltaLink
- Subsequent calls with deltaLink return only changes
- New messages: returned with full properties
- Moved/deleted: returned as `@removed` entries
- Read state changes: also returned (can be filtered out by application logic)

### Graph Change Notifications for Immediate Detection

Instead of polling, subscribe to change notifications:
```
POST /subscriptions
{
  "changeType": "created",
  "resource": "users/{sharedMailboxId}/mailFolders/Inbox/messages",
  "notificationUrl": "https://your-webhook.example.com/api/notifications",
  "expirationDateTime": "2026-04-12T00:00:00Z"
}
```

- Notification fires within seconds of email arrival
- Can include resource data (encrypted) to avoid a follow-up GET
- Eliminates the polling window entirely
- **Must** use application permissions for shared mailbox subscriptions
- **Must** implement lifecycle notifications for reliability
- Combine with periodic delta query catch-up (every 6-24 hours) for robustness

## 4. Concurrent Access Limits and Conflicts

### IMAP Connection Limits

| Limit | Value | Notes |
|-------|-------|-------|
| Concurrent IMAP connections per mailbox | ~8 | Not officially published for Exchange Online; derived from throttling policy |
| Per-user connection default (Exchange Server) | 16 | May differ for Exchange Online |
| Authentication | OAuth2 required | Basic auth permanently disabled |

### Graph API Throttling Limits

| Limit | Value | Notes |
|-------|-------|-------|
| MailboxConcurrency | 4 | Concurrent requests per app per mailbox |
| Request rate | 10,000 per 10 minutes | Per app per mailbox combination |
| Recommended rate | 4-10 requests/second | Microsoft's operational guidance |
| Global rate | 130,000 per 10 seconds | Per app across all tenants |
| Batch concurrency | 4 | Max concurrent requests within a batch targeting same mailbox |
| Change notification subscriptions | 1,000 | Per mailbox, across all applications |

### IMAP + Outlook/OWA Coexistence

- No documented conflicts between simultaneous IMAP and Outlook/OWA access
- All protocols see the same underlying mailbox state
- IMAP sessions consume connections from the per-mailbox pool
- Flag changes (including `\Seen`) propagate across all sessions
- Graph API requests are independent of IMAP connection limits (different throttling domain)

### Locking and Conflict Issues

- Exchange Online does **not** use pessimistic locking on messages
- Last-write-wins for property updates
- If poller and user simultaneously modify the same message (e.g., both set categories), last write wins
- No transactional guarantees across multiple message operations
- Delta queries and change notifications may experience brief delays (seconds, not minutes) in reflecting changes

## 5. Architectural Patterns for Coexistence

### Pattern A: Transport Rule + Dedicated Processing Mailbox (Recommended)

```
                    ┌─────────────────────┐
  Inbound Email ───>│  Exchange Transport │
                    │       Pipeline      │
                    └────────┬────────────┘
                             │
                    ┌────────┴────────┐
                    │  Mail Flow Rule  │
                    │  (BCC action)    │
                    └───┬─────────┬───┘
                        │         │
                        v         v
              ┌─────────────┐  ┌──────────────────┐
              │   Shared    │  │   Processing     │
              │   Mailbox   │  │   Mailbox        │
              │ (staff use) │  │ (bot-only)       │
              └─────────────┘  └────────┬─────────┘
                                        │
                                        v
                               ┌────────────────┐
                               │  Poller/Bot    │
                               │  (Graph API)   │
                               └────────────────┘
```

**How it works:**
1. Create a mail flow rule in Exchange Admin Center: "If recipient is shared-mailbox@domain.com, BCC to processing-mailbox@domain.com"
2. Bot monitors processing-mailbox exclusively via Graph API
3. Staff interact only with the shared mailbox
4. Zero interference between bot and staff

**Advantages:**
- Complete isolation: bot never touches the shared mailbox
- No race conditions with human users
- Bot can freely mark messages as read, delete them, etc. in its own mailbox
- No complex delta tracking needed; simple sequential processing
- Staff workflow completely unaffected

**Disadvantages:**
- Mail flow rule propagation delay: up to 30 minutes after creation
- BCC copy is a separate message (different Graph/IMAP IDs, though same Message-ID header)
- Additional mailbox to manage
- Rule only applies to messages processed through Exchange transport (not messages created directly in the mailbox via Graph API or drag-and-drop from another mailbox)
- Does not capture messages moved INTO the shared mailbox from other mailboxes by staff

**Implementation:**
```powershell
# PowerShell (Exchange Online)
New-TransportRule -Name "Copy to Processing Mailbox" `
  -RecipientAddressContainsWords "shared-mailbox@domain.com" `
  -BlindCopyTo "processing-mailbox@domain.com" `
  -Priority 0
```

### Pattern B: Graph API Observer on Shared Mailbox

```
              ┌─────────────────┐
              │   Shared        │
              │   Mailbox       │<──── Staff (Outlook/OWA)
              │                 │
              └────────┬────────┘
                       │
                       v
              ┌────────────────────┐
              │  Graph API Poller  │
              │  (observer only)   │──── External Database
              │                    │     (processed IDs)
              └────────────────────┘
```

**How it works:**
1. Bot uses Graph API change notifications (webhooks) for near-real-time detection
2. Periodic delta query catch-up (every 6-24 hours)
3. All messages tracked by `immutableId`
4. Processed messages marked with a category ("Bot-Processed") and/or extended property
5. External database stores all processed immutableIds

**Observer-only rules:**
- Bot NEVER modifies `isRead` state
- Bot ONLY adds categories (additive, does not remove existing categories)
- Bot uses `$select` to fetch only needed properties (reduces API calls and data transfer)
- Bot tolerates messages being moved or deleted by staff

**Implementation sketch:**
```python
# Pseudocode for Graph API observer pattern

# 1. Subscribe to new messages
subscription = graph.post("/subscriptions", {
    "changeType": "created",
    "resource": f"users/{SHARED_MAILBOX_ID}/mailFolders/Inbox/messages",
    "notificationUrl": WEBHOOK_URL,
    "expirationDateTime": expiry_time,
    "clientState": secret
})

# 2. On webhook notification, process message
def on_notification(message_id):
    msg = graph.get(
        f"/users/{SHARED_MAILBOX_ID}/messages/{message_id}",
        headers={"Prefer": 'IdType="ImmutableId"'},
        params={"$select": "id,subject,body,internetMessageId,receivedDateTime,from"}
    )
    
    if msg.immutableId in processed_db:
        return  # Already processed
    
    if contains_identifier_pattern(msg.body):
        ingest(msg)
        processed_db.add(msg.immutableId)
        
        # Optional: mark with category
        graph.patch(
            f"/users/{SHARED_MAILBOX_ID}/messages/{message_id}",
            {"categories": msg.categories + ["Bot-Processed"]}
        )

# 3. Periodic delta query catch-up
def delta_catchup():
    messages = graph.get(
        f"/users/{SHARED_MAILBOX_ID}/mailFolders/Inbox/messages/delta",
        headers={"Prefer": 'IdType="ImmutableId"'},
        params={"$deltatoken": last_delta_token}
    )
    for msg in messages:
        if "@removed" not in msg:
            on_notification(msg.id)
```

### Pattern C: Dedicated Subfolder with Inbox Rule

**How it works:**
1. Create a server-side Inbox rule: "If subject contains [IDENTIFIER], move to 'Bot-Intake' folder"
2. Bot monitors only the 'Bot-Intake' folder
3. Staff are instructed not to touch this folder

**Advantages:**
- Simpler than transport rules (user-level rule, not admin-level)
- Bot has a clean, isolated folder to process

**Disadvantages:**
- Inbox rules are less reliable than transport rules (can be disabled by users with mailbox permissions)
- Message is removed from Inbox (staff don't see it unless they check the subfolder)
- If staff need to see the message too, you need a more complex rule (copy, not move)
- Outlook Inbox rules cannot BCC; they can only move, copy, forward, or redirect

### Pattern D: Graph Change Notifications + Delta (No Polling)

This is the pure event-driven approach:

1. Subscribe to change notifications on `users/{id}/messages` (all messages in mailbox)
2. On `created` notification: check if message matches pattern, process, mark as processed
3. Delta query catch-up on a long interval (6-24 hours)
4. Track all processed messages by immutableId in external store

**This is the most responsive pattern** but requires:
- A publicly accessible webhook endpoint (HTTPS)
- Proper certificate management for encrypted notifications
- Lifecycle notification handling for subscription renewal
- Retry/catch-up logic for missed notifications

## 6. Exchange Online Specific Features

### Mail Flow Rules (Transport Rules)

- **Admin-level rules** processed by Exchange transport pipeline
- **BCC action**: delivers original to intended recipient(s) AND sends a copy to BCC target
- **Conditions**: recipient address, subject, headers, message properties, sender, etc.
- **Propagation time**: up to 30 minutes after creation/modification
- **Scope**: applies to all messages passing through Exchange transport (not internal moves)
- Cannot trigger on messages created directly in a mailbox (e.g., via Graph API `createMessage`)

### Journaling

- **Not viable for this use case**: Exchange Online journaling cannot deliver to an Exchange Online mailbox
- Requires on-premises or third-party archiving target
- Useful for compliance but not for real-time processing

### Compliance/Retention Features

| Feature | What it does | License requirement |
|---------|-------------|-------------------|
| Litigation Hold | Preserves all content including deleted items | Exchange Online Plan 2 |
| Retention Policies | Automated retention/deletion schedules | Included in most plans |
| Recoverable Items | Deleted items preserved for configurable period | Default 14 days, max 30 days |
| Microsoft Purview | Advanced compliance, eDiscovery | Microsoft 365 E5 or compliance add-on |

**For this use case:** Litigation Hold on the shared mailbox provides a safety net ensuring that even if staff delete an email before the poller processes it, the email is still recoverable from the Recoverable Items folder. However, the poller would need to scan Recoverable Items as well (accessible via Graph API).

### IMAP Authentication Requirements (Current as of 2026)

- **Basic authentication**: permanently disabled for IMAP in Exchange Online
- **OAuth2 required**: service principal with `IMAP.AccessAsApp` permission
- **Mailbox access**: granted via `Add-MailboxPermission` with `-AccessRights FullAccess`
- **Application access policy**: can restrict which mailboxes the app can access
- **SMTP AUTH basic auth**: ending April 30, 2026

### EWS Deprecation

- Exchange Web Services (EWS) will be **blocked** in Exchange Online starting **October 1, 2026**
- All new development should use Microsoft Graph API
- Existing EWS integrations must be migrated to Graph

## 7. Recommended Architecture

### For most scenarios: Pattern A (Transport Rule + Processing Mailbox)

This provides the strongest isolation and simplest implementation:

1. **Create a dedicated processing mailbox** (no license required for shared mailbox under 50GB)
2. **Create a transport rule**: BCC all inbound mail to the shared mailbox to the processing mailbox
3. **Build poller using Graph API** against the processing mailbox:
   - Use change notifications for near-real-time processing
   - Use delta queries as catch-up
   - Freely mark messages as read / move to "Processed" folder / delete after processing
4. **Apply Litigation Hold** to the shared mailbox as a safety net
5. **Track processed Message-IDs** in an external database for deduplication

### If transport rules are insufficient: Pattern B (Graph Observer)

Use when you need to process messages that arrive in the shared mailbox by means other than Exchange transport (e.g., messages created via Graph API, messages dragged from other mailboxes):

1. **Register a Graph API application** with `Mail.Read` and `Mail.ReadWrite` application permissions
2. **Restrict scope** using application access policies to limit access to the shared mailbox only
3. **Subscribe to change notifications** on the shared mailbox Inbox
4. **Use immutableId** for all message tracking
5. **Mark processed messages** with a category and/or extended property
6. **Never modify read state** or move messages
7. **Maintain external database** of processed immutableIds
8. **Run periodic delta query catch-up** every 6-12 hours
9. **Implement lifecycle notifications** for subscription health monitoring

### Critical Implementation Details

**For Graph API authentication:**
- Use client credentials flow (application permissions)
- Register app in Microsoft Entra ID
- Grant `Mail.Read` (or `Mail.ReadWrite` if marking categories) application permission
- Get tenant admin consent
- Use application access policies to restrict mailbox access

**For deduplication:**
- Primary key: `immutableId` (Graph API) or `internetMessageId` (cross-protocol)
- Store processed IDs in a persistent database (not in-memory)
- Periodically clean old entries (e.g., remove entries older than 90 days)

**For error handling:**
- Handle HTTP 429 (throttling) with exponential backoff using `Retry-After` header
- Handle subscription expiration with automatic renewal
- Handle lifecycle `subscriptionRemoved` and `missed` notifications
- Log all processing decisions for audit trail
