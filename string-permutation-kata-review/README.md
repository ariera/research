## Findings

### 1. Unicode string inputs can make enumeration fail

Severity: high

The README contract defines `seed: string`, `alphabet: set[char]`, and output as candidate strings. The implementation stores the alphabet as bytes and applies insert/delete/replace/swap at byte offsets.

Relevant code:

- `string-permutation-kata/rust/src/config.rs`: `alphabet: Arc<[u8]>`
- `string-permutation-kata/rust/src/mutations.rs`: loops over `0..seed.len()` and slices byte ranges
- `string-permutation-kata/rust/src/search.rs`: converts generated byte vectors with `String::from_utf8`

Repro:

```rust
let config = SearchConfig::new(
    "é",
    "é".as_bytes().to_vec(),
    1,
    1,
    KeyboardNeighbors::empty(),
)
.unwrap();

let result = enumerate_candidates(&config);
```

Actual result:

```text
Err("invalid utf-8 sequence of 1 bytes from index 0")
```

Why it happens:

- `"é"` is a valid Rust `String`, but it is two UTF-8 bytes.
- The delete operation can remove only one byte.
- That creates invalid UTF-8 candidates.
- `enumerate_candidates` aborts when converting the first invalid byte vector into `String`.

Impact:

- Valid documented string input can fail at runtime.
- The search is byte-edit distance, not character-edit distance.
- Completeness over `set[char]` is not guaranteed for non-ASCII input.

### 2. Invalid UTF-8 alphabet bytes are accepted at configuration time

Severity: high

`SearchConfig::new` accepts `Vec<u8>` for the alphabet and only checks that seed bytes are contained in that byte list. It does not reject bytes that can never form valid output strings.

Repro:

```rust
let result = SearchConfig::new("a", vec![b'a', 0xff], 1, 1, KeyboardNeighbors::empty());
```

Expected:

- Configuration should reject this input, or the public API should not promise `Vec<String>` output.

Actual:

- Configuration succeeds.
- Enumeration later fails once an inserted or replacement `0xff` candidate is emitted.

Impact:

- Invalid input is accepted too early.
- Callers get a late search-time error instead of a validation error.
- Some valid candidates may be discarded because the whole enumeration returns `Err`.

### 3. Documented enabled operations are not configurable

Severity: medium

The README contract lists `enabled operations` as an input. `SearchConfig` does not expose any operation flags, and `for_each_one_edit_neighbor` always generates all five operation classes: delete, insert, replace, keyboard-neighbor replace ranking, and adjacent swap.

Impact:

- A caller cannot ask for an insert-only, delete-only, no-swap, or no-arbitrary-replace search even though the contract says enabled operations are an input.
- Reachability and completeness are always calculated against the hard-coded operation set, which may disagree with the caller's intended search space.

## Recommended Fix Direction

Choose one contract and enforce it consistently:

1. Character-string API:
   - Change alphabet to a character collection such as `Vec<char>`.
   - Generate mutations over character indices, not byte offsets.
   - Keep `Vec<String>` output.

2. Byte-oriented API:
   - Rename/document the kata as byte-neighborhood enumeration.
   - Accept `seed: Vec<u8>` and return `Vec<Vec<u8>>`.
   - Keep UTF-8 conversion outside the core search.

For the documented kata, option 1 matches the current README best. Separately, add operation flags to `SearchConfig` or remove `enabled operations` from the contract.
